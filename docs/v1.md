# v1 — Scope & Definition of Done

**v1 = the engine, demo-able.** `charter + seed → a coherent, explorable browser world`, installable on any server with one command. No continuous agents yet (that's Phase B) — v1 proves the substrate.

## In scope

- **Charter schema** (Zod, versioned) including the divine-artifact document types (charter / master plan / decrees / amendments) — see [ADR-0001](adr/0001-charter-schema.md). Template ships as [`charters/_template/`](../charters/_template/).
- **World-data schemas**: chunk, manifest, asset-registry entry, NPC — including **behavior trees with a human-readable `label` on every node**.
- **Deterministic generation**: `charter + seed → skeleton world` (terrain, layout, prop placements from one licensed low-poly kit; license tracked in the asset registry).
- **`world-sim`**: world clock, behavior-tree runtime, WebSocket delta broadcast (compact state deltas at 1–5 Hz; clients interpolate).
- **`world-api`**: lore/read endpoints + the behavior-tree update endpoint (the L1 seam exists before agents use it).
- **Client**: chunk streaming, explore camera (drag / zoom / orbit), click-to-inspect panel (name, story, relationships from lore data + *current activity* from live behavior-tree state).
- **CI validation gate**: schema-valid · asset refs resolve · navmesh-lite connectivity · perf budget (poly / draw-call / chunk-size caps).
- **Deploy**: `docker compose up` → explorable world.
- **Example charters ×3** proving divergence (same engine, radically different worlds), plus the flagship world's real charter generated by a scripted **Founding Session** (the only model-in-the-loop piece of v1).

## Out of scope (Phase B+)

Continuous studio agents · orchestration control plane in the product loop · QA bot fleet · streaming/Twitch layer · viewer-wish pipeline · social automation · walk mode · player saves · deterministic client-side sim · cold-region archive/LOD.

## Definition of done — three acceptance tests

1. **Install:** fresh machine → `docker compose up` + a charter file → explorable, inspectable world in the browser in ≤ 15 minutes.
2. **Divergence:** swap the charter → a visibly, structurally different world from the same engine.
3. **The gate holds:** a PR containing an invalid chunk (bad asset ref, disconnected navmesh, blown perf budget) is rejected by CI.

Keep all three runnable — they are the v1 bar, not aspirations.

**Runnable:** `pnpm accept` ([`acceptance/run.mjs`](../acceptance/run.mjs)), summarized in CI on every push. A test **skips** (with the reason) while the piece it exercises hasn't landed, and **fails** if the piece exists but misses the bar. The v1 tag must pass `pnpm accept --no-skip` — no skips left.

- Test 1 contract (the deploy issue implements this): compose file at `deploy/compose.yaml`; the world counts as explorable when `GET :8080/` returns 200 and `GET :8080/world/manifest.json` returns the world manifest (port overridable via `FABLETON_PORT`). Timed against the 15-minute budget.
- Test 2 generates worlds from the template charter and the first committed charter under `charters/` (falling back to an engine fixture until the example charters land) and asserts the layout graph or dressing differs, with both worlds passing the gate.
- Test 3 breaks the sample world three ways at runtime (bad asset ref, disconnected navmesh, blown perf budget) and asserts the same `validate` command CI runs rejects each with the violation named — and that the pristine world passes.

## Build order

1. Charter schema + world-data schemas (**the keystone — everything else reads from these**)
2. CI validation gate (the gate teaches the standard; wire it before writing content)
3. Deterministic generation
4. `world-sim` + `world-api`
5. Client (streaming → explore camera → inspect)
6. Compose deploy
7. Example charters + Founding Session script
8. Acceptance harness green ×3
